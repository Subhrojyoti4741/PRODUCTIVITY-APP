FocusGuard - Smart Productivity & App Blocking System
Comprehensive Development Specifications for AI Developer

1. EXECUTIVE SUMMARY
Project Name: FocusGuard - Smart Productivity App with Intelligent App Blocking Platform: Android (Native or React Native/Flutter) Core Purpose: A productivity application that uses time-based app blocking, task completion incentives, and curated content access to eliminate distractions and boost focus. 

2. CORE FEATURES SPECIFICATION
2.1 Midnight Lockdown System
Functionality: 
Automatically blocks all apps except Phone and Email at midnight (12:00 AM)
Users must complete their to-do list to unlock all apps
Provides emergency override with consequences (XP penalty, streak break)
 Technical Requirements: 
Use Android AccessibilityService or Device Administrator API for app blocking
Implement background service that monitors time and task completion status
Whitelist: com.android.phone (Phone), com.google.android.gm (Gmail), com.samsung.android.email (Samsung Email)
Block mechanism: Overlay blocker screen or use AppOpsManager to restrict app launches
Store block schedule in local SQLite database with encrypted preferences
2.2 Task-Based App Unlocking System
Functionality: 
Users create daily to-do lists with time allocations
During scheduled task time, only specified apps are allowed (e.g., "Study Math" allows Calculator, Notion, YouTube edu playlist)
All other apps remain blocked until task is marked complete or time expires
Users earn XP points for completing tasks on time
 Technical Requirements: 
Task Model: task_id, title, description, start_time, end_time, allowed_apps[], xp_reward, status (pending/active/completed/failed)
Foreground service monitors current active task and blocks unauthorized app launches
Use UsageStatsManager to detect app switching attempts
Implement task timer with notifications (5 min before end, task complete reminder)
Sync tasks with Google Calendar API (optional)
2.3 Curated YouTube Learning Mode
Functionality: 
Users sync YouTube playlists or paste video URLs
In-app embedded YouTube player displays only curated videos
All other apps (including native YouTube) remain blocked during learning session
Prevents algorithm-driven recommendations and infinite scroll
Tracks watch time and completion percentage
 Technical Requirements: 
Integrate YouTube Data API v3 for playlist fetching
Use YouTube Android Player API or WebView with embedded iframe player
Disable autoplay, related videos, and comments in embedded player
Video Model: video_id, title, thumbnail_url, duration, watch_progress, playlist_id, date_added
Block com.google.android.youtube package while in learning mode
Allow users to manually add URLs with regex validation (youtube.com/watch?v=...)

4. BACKEND & DATA ARCHITECTURE
4.1 Database Schema (SQLite)
Table 1: users 
 - user_id (INTEGER PRIMARY KEY AUTOINCREMENT) - username (TEXT NOT NULL) - email (TEXT UNIQUE) - profile_picture_url (TEXT) - level (INTEGER DEFAULT 1) - total_xp (INTEGER DEFAULT 0) - current_streak (INTEGER DEFAULT 0) - longest_streak (INTEGER DEFAULT 0) - created_at (TIMESTAMP) - last_login (TIMESTAMP) 
Table 2: tasks 
 - task_id (INTEGER PRIMARY KEY AUTOINCREMENT) - user_id (INTEGER FOREIGN KEY) - title (TEXT NOT NULL) - description (TEXT) - start_time (TIMESTAMP NOT NULL) - end_time (TIMESTAMP NOT NULL) - category (TEXT) - allowed_apps (TEXT) -- JSON array of package names - xp_reward (INTEGER) - status (TEXT) -- 'pending', 'active', 'completed', 'failed', 'skipped' - repeat_pattern (TEXT) -- 'once', 'daily', 'weekdays', JSON for custom - created_at (TIMESTAMP) - completed_at (TIMESTAMP) 
Table 3: youtube_videos 
 - video_id (TEXT PRIMARY KEY) - user_id (INTEGER FOREIGN KEY) - playlist_id (TEXT) - title (TEXT) - thumbnail_url (TEXT) - duration (INTEGER) -- seconds - watch_progress (INTEGER) -- seconds watched - is_completed (BOOLEAN) - date_added (TIMESTAMP) 
Table 4: app_usage_logs 
 - log_id (INTEGER PRIMARY KEY AUTOINCREMENT) - user_id (INTEGER FOREIGN KEY) - package_name (TEXT) - app_name (TEXT) - usage_date (DATE) - total_time_minutes (INTEGER) - open_count (INTEGER) 
Table 5: achievements 
 - achievement_id (INTEGER PRIMARY KEY AUTOINCREMENT) - user_id (INTEGER FOREIGN KEY) - achievement_type (TEXT) -- 'first_task', 'week_streak', 'productivity_champ', etc. - title (TEXT) - description (TEXT) - icon_url (TEXT) - earned_at (TIMESTAMP) 
Table 6: app_block_schedule 
 - schedule_id (INTEGER PRIMARY KEY AUTOINCREMENT) - user_id (INTEGER FOREIGN KEY) - rule_name (TEXT) -- 'Midnight Lockdown', 'Study Session', etc. - start_time (TIME) - end_time (TIME) - days_of_week (TEXT) -- JSON array ['MON', 'TUE', ...] - blocked_apps (TEXT) -- JSON array or '*' for all except whitelist - whitelist_apps (TEXT) -- JSON array of allowed apps - is_active (BOOLEAN) 
4.2 Background Services & Workers
Service 1: TaskMonitorService (Foreground Service) 
Purpose: Monitors current time and active tasks, enforces app blocking rules
Runs: Always (with foreground notification showing current task or "No active tasks")
Actions: Every 5 seconds, check if current time matches any task schedule; if blocked app is launched, immediately show blocker overlay; update task status (active/completed/failed); send notifications for task start/end
 Service 2: MidnightLockdownScheduler (WorkManager) 
Purpose: Schedules daily midnight app lockdown
Runs: Once per day at 12:00 AM
Actions: Enable global app block mode; send notification "Midnight lockdown active - complete tasks to unlock"; persist state to SharedPreferences
 Service 3: UsageStatsCollector (WorkManager - Periodic) 
Purpose: Collects daily app usage statistics
Runs: Every 1 hour
Actions: Query UsageStatsManager for app usage data; store in app_usage_logs table; update screen time widget
 Broadcast Receivers: 
BOOT_COMPLETED: Restart TaskMonitorService after device reboot
PACKAGE_ADDED/REMOVED: Update installed apps list for blocker selection
SCREEN_ON/OFF: Pause/resume usage tracking
4.3 Android Permissions & APIs Required
Critical Permissions: 
android.permission.PACKAGE_USAGE_STATS (Usage Access)
android.permission.BIND_ACCESSIBILITY_SERVICE (Accessibility Service)
android.permission.FOREGROUND_SERVICE
android.permission.POST_NOTIFICATIONS (Android 13+)
android.permission.INTERNET (YouTube API)
android.permission.REQUEST_IGNORE_BATTERY_OPTIMIZATIONS
android.permission.SYSTEM_ALERT_WINDOW (Draw overlay)
 Key APIs: 
AccessibilityService: Detect app launches and intercept with blocker overlay
UsageStatsManager: Retrieve app usage data for statistics
WindowManager: Display blocker overlay on top of blocked apps
WorkManager: Schedule background tasks (lockdown, usage collection)
NotificationManager: Send task reminders and achievement notifications
YouTube Data API v3: Fetch playlists and video metadata
YouTube Android Player API: Embed and control video playback

5. DEVELOPMENT APPROACH & MILESTONES
5.1 Technology Stack Recommendation
Option A: Native Android (Kotlin) 
Pros: Best performance, full access to Android APIs, superior app blocking control
Cons: Android-only, longer development time
Use: Jetpack Compose for UI, Kotlin Coroutines, Room Database, WorkManager
 Option B: React Native 
Pros: Cross-platform (Android/iOS future), faster development, reusable web dashboard code
Cons: Limited access to Android system APIs (need native modules for app blocking)
Use: Expo or bare React Native, React Navigation, AsyncStorage, react-native-app-usage
 Recommendation: Native Android (Kotlin) for MVP due to critical app blocking requirements. Port to React Native later if cross-platform is needed.

6. SECURITY & PRIVACY CONSIDERATIONS
Data Privacy: 
All user data stored locally by default (no cloud unless opted in)
Encrypt sensitive data (passwords if local auth, API tokens) using Android Keystore
No personally identifiable information collected without explicit consent
Comply with GDPR/CCPA for data export and deletion
 Permissions Transparency: 
Clearly explain why each permission is needed in onboarding
Provide in-app guides for granting usage access and accessibility permissions
Never request permissions not essential to core functionality
 YouTube API Compliance: 
Follow YouTube API Terms of Service (no downloading videos, respect rate limits)
Display YouTube branding as required by API guidelines
Implement exponential backoff for API requests
 App Blocking Security: 
Prevent app uninstallation during active blocking period (optional, warn user)
Emergency override should have cooldown period (e.g., 1 override per day max)
Log all override attempts for user transparency
